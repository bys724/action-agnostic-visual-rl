## ⚠️ 청구 단위: 일 단위 올림 (CEIL)

**모든 컴퓨트 자원은 실제 사용 시간을 일 단위로 올림(ceil)해서 청구된다.**

```
청구일수 = ceil(실제_사용_시간 / 24h)
청구액   = 단가 × 자원 수 × 청구일수
```

| 실제 elapsed | 청구 일수 | 1 H100 비용 | 8 H100 비용 |
|-------------|----------|------------|------------|
| 30분 | 1일 | 61,000 | 488,000 |
| 1시간 | 1일 | 61,000 | 488,000 |
| 23시간 | 1일 | 61,000 | 488,000 |
| 24시간 정확히 | 1일 | 61,000 | 488,000 |
| 24시간 1분 | **2일** | **122,000** | **976,000** |
| 47시간 | 2일 | 122,000 | 976,000 |
| 48시간 1분 | **3일** | **183,000** | **1,464,000** |
| 71시간 | 3일 | 183,000 | 1,464,000 |
| 73시간 | **4일** | **244,000** | **1,952,000** |

**전략 (낭비 방지)**:
1. **짧은 sanity test에 GPU 절대 안 씀** — 1시간 테스트도 1일 청구. 가능하면 CPU 인터랙티브로 대체
2. **잡 시간은 24h의 배수 직전에 끝내도록 설계** — `--time=23:30:00`, `--time=47:30:00`, `--time=71:30:00`
3. **--time을 24:00:00으로 두면 안 됨** — 1분만 초과해도 2일 청구. **23시간대로 설정**
4. **여러 작업 묶기** — 같은 sbatch 안에서 sequential로 묶으면 elapsed 한 번. 단 sbatch당 ceil인지 calendar day 기반인지 운영팀 확인 필요
5. **빠른 디버그 사이클은 로컬 워크스테이션 활용** — 클러스터는 검증된 잡만

⚠️ **불확실 사항** (운영팀 확인 권장):
- "1일"이 (a) 잡 elapsed 24h 단위 ceil인지, (b) calendar day(자정 기준) 단위인지
- (b)라면 23:30 시작해서 다음날 01:30 종료 = 2일 청구 가능 → 잡 시작 시간도 신경 써야 함

## 기록 절차

### 1. 잡 제출 시
즉시 "진행 중" 표에 행 추가:
```
| <JobID> | YYYY-MM-DD HH:MM | <파티션> | <노드>N × <자원> | <한 줄 목적> |
```

### 2. 잡 종료 확인 시
sacct로 정확한 시각 조회 → "진행 중"에서 "완료" 표로 이동:
```bash
sacct -j <JobID> --format=JobID,JobName,Partition,AllocTRES,Start,End,Elapsed,State
```
"완료" 표에 기록할 때 **비용 추정**도 계산해서 기입 (위 단가 표 참조).

### 3. 저장소 증설 시에만
mrg 그룹은 이미 10 TB를 할당받았으므로 그 범위 내 사용은 추적 불필요.
**10 TB 초과 증설을 신청한 경우에만** "저장소" 표에 1줄 추가.

### 4. 다중 작업 묶음
sbatch array는 부모 JobID로 1줄 + 비고에 array 범위 (예: `array=1-5`).
잡 단계가 많은 워크플로우는 단계별로 분리 기록.

### 5. 비정상 종료
`TIMEOUT`, `FAILED`, `CANCELLED`, `OUT_OF_MEMORY` 등은 State에 명시 + 비고에 짧은 원인.
다음 잡에서 같은 실수 반복 방지.
